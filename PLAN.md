# PLAN.md - Phase 별 개발 목표

> 근거 문서: [AGENTS.md](./AGENTS.md), [PRD.md](./PRD.md), `docs/FEATURES/*.md`
> 각 Phase의 상세 설계는 `docs/design/phaseN.md`에서 확인할 수 있다.

## 이 문서의 성격

- 각 Phase는 **일정(소요 시간)을 포함하지 않는다.** 오직 "무엇을 만들고, 무엇을 확인해야 하는가"만
  다룬다.
- 각 Phase가 끝나면 **실제로 동작하는 소프트웨어**가 나와야 한다. 미완성/스텁 상태로 다음
  Phase로 넘어가지 않는다.
- 각 Phase 종료 시, **고객(사람)이 콘솔을 직접 실행하여 해당 Phase에서 추가된 기능을 눈으로
  확인하고 피드백을 줄 수 있어야 한다.**
- 아래 "고객님이 확인할 부분"은 Phase별로 실행해보고 중점적으로 봐야 할 포인트를 안내한다.

---

## Phase 1. 프로젝트 골격 + 시료 관리

**목표 기능**: [시료 관리](./docs/FEATURES/sample-management.md), 메인 메뉴 뼈대

- MVC(Model/Controller/View) 구조와 JSON 파일 기반 데이터 영속성의 기본 골격을 구성한다.
- 콘솔 실행 시 메인 메뉴가 뜨고, "시료관리" 메뉴로 진입할 수 있다.
- 시료 등록 / 시료 조회(재고 포함) / 시료 이름 검색 기능이 동작한다.

**고객님이 확인할 부분**:
- 프로그램을 실행하면 메인 메뉴가 정상적으로 뜨는지
- 새 시료를 등록할 수 있는지 (시료ID, 이름, 평균 생산시간, 수율 입력)
- 등록한 시료가 목록 조회에 나오는지, 초기 재고가 0으로 보이는지
- 이름으로 검색했을 때 원하는 시료가 잘 나오는지
- 중복된 시료ID로 등록을 시도했을 때 거부되는지
- **프로그램을 종료했다가 다시 실행해도 등록했던 시료가 그대로 남아있는지** (데이터 저장 확인)

---

## Phase 2. 시료 주문 + 주문 승인/거절

**목표 기능**: [시료 주문](./docs/FEATURES/sample-order.md),
[주문 승인/거절](./docs/FEATURES/order-approval-rejection.md)

- 고객 주문을 접수(예약)하면 `RESERVED` 상태로 생성된다.
- 재고가 충분한 주문을 승인하면 즉시 `CONFIRMED`로 전환되고 재고가 차감된다.
- 재고가 부족한 주문을 승인하면 `PRODUCING`으로 전환된다 (실제 생산 처리는 Phase 3에서 완성).
- 주문을 거절하면 `REJECTED`로 전환된다.

**고객님이 확인할 부분**:
- Phase 1에서 등록한 시료로 주문을 넣었을 때 상태가 `RESERVED`로 뜨는지
- 재고가 충분한 상태에서 주문을 승인했을 때, 주문이 `CONFIRMED`로 바뀌고 재고가 그만큼
  줄어드는지
- 재고가 부족한 상태에서 주문을 승인했을 때, 주문이 `PRODUCING`으로 바뀌는지 (이 단계에서는
  아직 생산이 실제로 진행되지 않아도 됨 — Phase 3에서 확인)
- 주문을 거절했을 때 `REJECTED`로 바뀌고, 이미 처리된 주문을 다시 승인/거절하면 거부되는지

---

## Phase 3. 생산라인

**목표 기능**: [생산라인](./docs/FEATURES/production-line.md)

- Phase 2에서 재고 부족으로 `PRODUCING` 상태가 된 주문이 생산 큐(FIFO)에 등록되어 실제로
  처리된다.
- 생산 큐, 실생산량(`ceil(부족분/수율)`), 총생산시간(`평균생산시간 × 실생산량`)을 조회할 수
  있다.
- 생산이 완료되면 주문 상태가 `PRODUCING -> CONFIRMED`로 바뀌고, 생산된 만큼 재고에 반영된다.

**고객님이 확인할 부분**:
- 재고가 부족해 `PRODUCING`이 된 주문이 생산라인 메뉴의 대기 목록에 나오는지
- 여러 건을 동시에 넣었을 때 먼저 등록한 순서(FIFO)대로 처리되는지
- 생산 현황(현재 생산 중인 시료, 진행 상황 등)이 화면에 표시되는지
- 생산이 끝나면 해당 주문이 `CONFIRMED`로 바뀌고, 재고 수량이 실제로 늘어나는지
- (참고용) 실생산량/총생산시간 숫자가 수율/평균생산시간 기준으로 납득 가능한 값인지

---

## Phase 4. 출고처리

**목표 기능**: [출고처리](./docs/FEATURES/shipment.md)

- `CONFIRMED` 상태 주문을 선택해 출고 처리를 실행하면 `RELEASE`로 전환된다.

**고객님이 확인할 부분**:
- `CONFIRMED` 상태 주문을 출고 처리했을 때 `RELEASE`로 바뀌는지
- `CONFIRMED`가 아닌 주문(예: `RESERVED`, `PRODUCING`, 이미 `RELEASE`된 주문)을 출고 처리하려고
  하면 거부되는지
- 이 시점에서 **주문 등록 -> 승인 -> (필요 시 생산) -> 출고**까지 전체 흐름을 한 번씩 직접
  진행해보고 문제가 없는지 확인

---

## Phase 5. 모니터링

**목표 기능**: [모니터링](./docs/FEATURES/monitoring.md)

- 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE`) 주문 현황을 확인할 수 있다
  (`REJECTED`는 제외).
- 시료별 재고 수량과 "여유/부족/고갈" 상태를 확인할 수 있다.

**고객님이 확인할 부분**:
- 지금까지(Phase 1~4) 만들어 둔 주문/시료 데이터를 기준으로, 모니터링 화면의 숫자가 실제
  상태와 맞는지
- `REJECTED` 처리한 주문이 모니터링 집계에서 빠지는지
- 재고가 넉넉한 시료는 "여유", 빠듯한 시료는 "부족", 0개인 시료는 "고갈"로 정확히 표기되는지

---

## Phase 6. 통합 도구 및 전체 마무리

**목표 기능**: 데이터 모니터링 도구, Dummy 데이터 생성 도구, 전체 시나리오 통합, 문서/품질 정리
(참고: [CLAUDE.md](./CLAUDE.md) 참고 PoC 저장소)

- 저장된 데이터를 콘솔에서 실시간으로 조회하는 데이터 모니터링 도구를 추가한다.
- 테스트용 더미 데이터를 생성해 시료/주문 데이터를 빠르게 채울 수 있는 도구를 추가한다.
- 시료등록 → 주문 → 승인/거절 → 생산 → 출고 → 모니터링까지 전체 시나리오가 한 프로그램 안에서
  끊김 없이 동작하는지 통합적으로 점검하고, 문서(CLAUDE.md/PRD.md/docs)와 코드 품질(Clean
  Code, 테스트 커버리지)을 최종 정리한다.

**고객님이 확인할 부분**:
- 더미 데이터 생성 도구로 데이터를 채운 뒤, 처음부터 끝까지 전체 업무 흐름을 막힘 없이 진행해
  볼 수 있는지
- 데이터 모니터링 도구로 현재 저장된 데이터 상태를 실시간으로 들여다볼 수 있는지
- 지금까지 Phase별로 지적했던 피드백/버그가 모두 반영되었는지 최종 확인
